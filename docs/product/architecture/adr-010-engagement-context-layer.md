# ADR-010: Engagement Context Layer — Third Deployment Pattern

**Status**: Accepted
**Date**: 2026-05-14
**Scope**: Slice 05 (coach-buddy-slice-05) — persistent engagement context across sessions
**Extends**: ADR-008 (portable install two-layer model)

---

## Context

ADR-008 introduced the two-layer portable install: `custom-instructions.md` as the always-on ambient layer, `SKILL.md` in Project Knowledge as the invocable thinking-partner pipeline. This solved the context-switching problem (J3) and the distribution problem (J5).

A new pattern emerged from real use: coaches running ongoing engagements need more than a thinking-partner available in context — they need the tool to carry knowledge of the team across sessions. Each conversation currently starts cold. Hypotheses evaporate. Interventions are not tracked. The coaching arc is invisible.

Two structural gaps drive this:

1. **Statelesness**: Claude has no memory between conversations. Every session begins without knowledge of the team, prior observations, or what has been tried.
2. **Work-as-Imagined drift**: Without a current picture of the team's work state, the coach reasons from memory — which is Work-as-Imagined, not Work-as-Done.

The engagement context layer addresses both gaps through persistent files, not through model memory.

---

## Decision

### Third deployment pattern: Engagement Context Layer

This is an additive extension to the existing two patterns:

| Pattern | What it provides | When to use |
|---------|-----------------|-------------|
| Dedicated coaching project (Slices 01-02) | Full pipeline, always-on | One project per coaching practice |
| Portable team project install (Slice 03, ADR-008) | Pipeline on demand in any team project | Multi-team coaches who work inside team projects |
| **Engagement context layer (Slice 05, this ADR)** | **Persistent grounding across sessions** | **Any ongoing engagement where context accumulates** |

The third pattern can be layered on top of either of the first two. It does not replace them.

### File layout

**In the coach-buddy package (repo):**

```
skills/
  cb-init/SKILL.md       — engagement scaffolding
  cb-log/SKILL.md        — coaching log capture
  cb-retro/SKILL.md      — retro action tracking
  cb-snapshot/SKILL.md   — board snapshot
references/
  coaching-practice/
    coaching-log-format.md
    board-snapshot-guide.md
```

**In the coach's project (user data, not in package):**

```
engagements/
  <team-slug>/
    config.json          — engagement registry (tool type, project key, path)
    CONTEXT.md           — static team knowledge
    COACHING_LOG.md      — Safety-II observations, most-recent-first
    RETRO_ACTIONS.md     — retro action tracker
    HISTORY.md           — team lineage and structural changes
    snapshots/
      YYYY-MM-DD-board.md
```

### config.json as engagement registry

`cb-init` writes a `config.json` to `engagements/<team-slug>/`. All downstream skills (`cb-log`, `cb-retro`, `cb-snapshot`) read it on invocation. This eliminates re-prompting between skills and makes the engagement path explicit.

Schema (v1):
```json
{
  "version": 1,
  "engagement": { "name": "...", "slug": "...", "created": "YYYY-MM-DD" },
  "tool": { "type": "jira|linear|shortcut|none", "project_key": "...", "board_id": "...", "wip_age_threshold_days": 5 }
}
```

### Safety-II-informed COACHING_LOG.md

The COACHING_LOG.md format is structurally Safety-II-informed. This is not cosmetic:

- **Observed** captures Work-as-Done (what actually happened), not Work-as-Imagined (what should have happened)
- **Hypothesis** is explicitly testable (If/Then format), not a judgement
- **Intervention** is named, enabling cross-session comparison
- **Pattern/Signal** is tentative — the format resists premature closure

The Safety-II framing is encoded in `references/coaching-practice/coaching-log-format.md` so coaches understand why the structure exists, not just how to fill it in.

### Tool-agnostic board snapshot

`cb-snapshot` isolates tool-specific query logic in named sections (`## Tool: Jira`, `## Tool: none`). The Jira JQL implementation is the reference. When `tool.type = "none"`, the skill accepts a manual paste and structures it into the same four-section format. New tool support requires only a new named section — no structural change to the skill.

### Chat project knowledge sync pattern

The engagement layer bridges Claude Code (live skills) and Claude Chat (static project knowledge):

1. Coach runs skills in Claude Code → files accumulate in `engagements/<team-slug>/`
2. Before a Claude Chat session, coach uploads CONTEXT.md, COACHING_LOG.md (or a selected portion), and the latest snapshot to Chat project knowledge
3. Chat session has the same grounding as a Claude Code session

The snapshot file is the key bridge artefact: it is generated by a live skill but readable as static context.

---

## Naming convention

All engagement skills use the `cb-` prefix: `/cb-init`, `/cb-log`, `/cb-retro`, `/cb-snapshot`. This mirrors the `nw-` prefix convention from nWave and prevents namespace collisions with other tools in the same skills directory. The core thinking-partner skill (`/coach-buddy`) predates this convention and retains its full name.

## Alternatives Considered

### Alternative A: Dedicated "engagement mode" in SKILL.md

Add engagement awareness directly to SKILL.md. On invocation, it reads COACHING_LOG.md and CONTEXT.md automatically.

**Why rejected**: This couples engagement ops to the thinking-partner pipeline. A coach who hasn't set up an engagement would see prompts or errors from engagement-aware code on every `/coach-buddy` invocation. The two concerns (thinking-partner quality, engagement persistence) are independent — they should be independently installable and activatable.

### Alternative B: External memory service or MCP

Use an MCP memory server (e.g. a vector store or external notes tool) to persist coaching observations.

**Why rejected**: Introduces a runtime dependency not controllable by the coach. The markdown-on-filesystem approach has zero dependencies, works offline, is human-readable, is version-controllable, and is uploadable to Chat project knowledge. For a tool where transparency is a quality attribute (ADR-001), visible files beat hidden memory.

### Alternative C: Single `team-context.md` file (existing README pattern)

The README already documents a "persistent team context" pattern using a single `team-context.md` file. This could be extended rather than replaced.

**Why extended rather than replaced**: `team-context.md` is a monolithic snapshot that the coach maintains manually. The engagement context layer adds structured capture (`cb-log`, `cb-retro`) and automated snapshot (`cb-snapshot`) on top. The CONTEXT.md in the new pattern is equivalent to `team-context.md` for the static knowledge portion. The engagement layer supersedes the single-file pattern for ongoing engagements while remaining compatible with it.

## Consequences

**Positive:**
- Persistent coaching arc across sessions — coaching builds rather than starts over
- Work-as-Done grounding via snapshots — reasoning from evidence, not memory
- Chat + Code coherence — same files usable in both interfaces
- Zero runtime dependencies for cb-init, cb-log, cb-retro — pure file operations
- Each skill independently installable — coaches can adopt incrementally

**Negative / Watch items:**
- Engagement files are not automatically committed or backed up — coaches who work in a project without version control may lose data. Mitigation: README recommends initialising a git repo in the project.
- config.json ties skills to a single engagement slug per session — coaches managing multiple simultaneous team engagements need to specify the slug explicitly or run skills from different project directories. Document this clearly in README.
- board snapshot requires MCP for automated query — without a Jira MCP, the manual paste path is the only option. Mitigation: manual paste is designed as a first-class path, not a fallback.
- CONTEXT.md is manually populated — if the coach never fills it in, the engagement layer delivers less value. Mitigation: placeholder sections are designed as compelling prompts.

### Post-deployment watch items (30-day review)

- **Adoption rate**: How many coaches reach `cb-log` after `cb-init` (i.e. do they actually use the log)?
- **Quick-capture vs full-entry ratio**: Are coaches using quick capture primarily, or filling all fields? If mostly quick-capture, the "to fill" scaffolding may be good enough; if full entries dominate, the quick-capture path may be unnecessary overhead.
- **CONTEXT.md fill rate**: What proportion of CONTEXT.md sections are populated after 30 days? If fewer than half, the placeholder design needs work.
- **Chat sync adoption**: Do coaches upload files to Chat project knowledge? If not, the bridge pattern is not being used — investigate whether it's discoverability or friction.
