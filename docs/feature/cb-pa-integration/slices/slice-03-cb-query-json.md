# Slice 03 — cb-query --format json

**Feature**: cb-pa-integration  
**Story**: US-003  
**Estimate**: ≤4h  
**Status**: pending

## Goal

Add `--format json` to `cb-query`, emitting a structured JSON response conforming to the PA contract (v1.0.0-draft, with DW-2 scoping on signal_summary). Unblocks PA Slices 2–4 and validates DW-2 hypothesis.

## IN Scope

- `--format json` flag on cb-query skill
- JSON response schema per PA contract field table
- `status: ok | degraded | error`
- `degraded` path: board MCP unavailable → `wip_aged: []`, `warnings` populated
- `error` path: engagement folder not found → minimal response
- `signal_summary` scoped to engagement-health domain (DW-2)
- All fields sufficient for PA-side next-action prioritisation

## OUT Scope

- signal_type tagging on open_actions (PA contract Section 3 — future)
- Multi-slug batch query
- PA-side synthesis logic
- Dashboard rendering

## Learning Hypothesis (DW-2)

**Disproves**: `signal_summary` generated by coach-buddy creates duplication or confusion when PA combines it with calendar/chat signals.  
**Confirms**: Scoped engagement-health summary is a useful atomic unit — displayable standalone when other sources are unavailable, naturally composable when they are.

**Dogfood test**: PA displays `signal_summary` in ≥1 real pre-session brief without modification. If PA overrides or ignores it, DW-2 is falsified → revisit in next iteration.

## Acceptance Criteria

- JSON conforms to PA contract schema with DW-2 scoping on signal_summary
- `status: ok` — all required fields present
- `status: degraded` — `wip_aged: []`, `warnings` array populated, other fields present
- `status: error` — `team` and `error` present; no other fields required
- `open_actions[].evidenced`, `last_capture`, `open_hypotheses[].status`, `wip_aged[].age_days` all present on ok response
- `--format json` absent: Slice 02 human-readable behaviour unchanged
